## Executive Summary

HerbalBookForge is the herbal-themed baseline authoring application and remains the visual reference for shared shell quality, while preserving domain-specific herbal workflows and governance traceability.

## Unified LLM API and State Architecture (v0.9.6, 2026-04-25)

- HBF.UNI1: All LLM API calls SHALL be routed through a single unified function (callLlmAgent) that takes agent, context, userPrompt, and response parser as arguments.
- HBF.UNI2: Each tab SHALL prepare its own context and parse the LLM response as needed, but SHALL always update the central project JSON state and re-render the UI from this state.
- HBF.UNI3: The UI SHALL always render from the central project JSON state after any LLM call or user edit.

 




HBF.UNI4: Every LLM API call SHALL use the system prompt (agent prompt) corresponding to the function/agent being exercised (Book Goals Agent, Outliner, Chapter Annotator, Drafter, Safety, etc.).

HBF.UNI4: If an agent prompt is missing, a default agent prompt SHALL be used for that agent type.

HBF.UNI4: If an unknown agent type is requested, the app SHALL throw an error and not make the LLM call.

HBF.UNI4: This mapping SHALL be enforced in code and traceable to requirements.

HBF.UNI4.1: All agent prompts (Book Goals, Outliner, Chapter Annotator, Safety) SHALL require strict JSON output for all fields that need to be parsed. All response parsing logic SHALL expect and validate this format, with fallback/error handling for malformed responses. This improves reliability, testability, and downstream automation.

## Functional Requirements

### Chapter Annotation String Consistency (v0.9.6, 2026-04-25)

- HBF.CHO7: All chapter annotation context, storage, and UI rendering SHALL always be a string, never an object or [object Object]. This applies to all flows: generation, regeneration, save, consistency edit, and UI. Any non-string annotation SHALL be converted to a string before being sent to the LLM, saved in project state, or rendered in the UI.
 


### Chapter Outlines Tab Requirements (Sprint X)

HBF.BG.G2: Accept Goals SHALL generate an initial outline using the LLM (never static) when no outline is currently present. The outline SHALL depend on the book goals and be generated by the LLM with reasoning about how to meet those goals.

HBF.CHO3: User SHALL be able to edit each chapter outline inline and explicitly save changes.

HBF.CHO4: Chapter Outlines Tab SHALL support per-chapter regeneration via the Chapter Annotator Agent.

HBF.CHO5: When the last chapter outline is created, the app SHALL resubmit all chapter outlines to the Chapter Annotator Agent for a second-pass consistency-editing step, harmonizing style, structure, and cross-references across chapters.
